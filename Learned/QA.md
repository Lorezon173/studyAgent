## 阶段 1.1-1.2 小测验

### Q1：为什么 API 边界用 Pydantic、内部 state 用 TypedDict？

**来源阶段**：1.2

**用户回答**：
> API 的边界有更加严格的生产条件限制，需要对数据类型验证，保证数据的可靠性，而 state 内部只有 agent 的状态流转和信息交流无需额外验证，反而对内存占用、响应时间要求更高，因此用静态的 TypedDict 更合适。

**评价**：

| 论点 | 评价 | 说明 |
|---|---|---|
| API 需要数据验证保证可靠性 | ✅ 正确 | [schemas.py:5](../app/models/schemas.py#L5) `session_id: str = Field(...)` — `...` 表示必填 |
| 内部无需额外验证 | ✅ 正确 | 节点之间是可信代码 |
| 内存占用、响应时间 | ⚠️ 方向对但不是核心 | TypedDict 确实零开销，但真正原因是 LangGraph 合并语义 |

**总分**：优秀（核心论点全对）

**补充要点**：
- 遗漏关键原因：Pydantic `BaseModel` 做 LangGraph state 时，嵌套字段合并会覆盖整个子结构（不可变语义）；TypedDict 本质是 `dict`，LangGraph 对它做字段级 merge，天然适配。这是**架构适配**问题，不仅是性能。

**详细扩展**：
> 见 [top-知识扩展.md#pydantic-vs-typeddict-选型全景](top-知识扩展.md#pydantic-vs-typeddict-选型全景)

### Q2：`total=False` 在 `LearningState` 里起什么作用？

**来源阶段**：1.2

**用户回答**：
> 表示所有字段是可选的，即每个状态机都可以按照自己的需要修改自己相关的部分，更高效、可维护性更好。

**评价**：

| 论点 | 评价 | 说明 |
|---|---|---|
| 所有字段可选 | ✅ 正确 | [state.py:17](../app/agent/state.py#L17) `class LearningState(TypedDict, total=False)` |
| 每个节点按需读写 | ✅ 正确 | 如 `diagnose_node` 只返回 `{"diagnosis": "...", "stage": "diagnosed"}` |
| 更高效、可维护性更好 | ✅ 正确 | 节点只关心自己那几个字段，改动互不干扰 |

**总分**：优秀（无遗漏，回答准确）

**补充要点**：
- 无遗漏。可以额外记住：如果不用 `total=False`，每个字段都需要标 `Optional`，但 TypedDict 的 `Optional` 只影响类型检查器，运行时不会校验——所以 `total=False` 是更简洁的做法。

### Q3：跨周持续教学选哪种方案？为什么？

**来源阶段**：1.2

**用户回答**：
> Temporal 方法（需要详细介绍 Temporal）

**评价**：

| 论点 | 评价 | 说明 |
|---|---|---|
| 选择 Temporal | ✅ 正确 | Temporal 的 Workflow 持久化 + Activity 重试机制是跨周长任务的最佳选择 |
| 要求详细介绍 | ✅ 主动求知 | 正确识别知识盲区并主动提问 |

**总分**：正确（方案选择准确，主动求知是加分行为）

**补充要点**：
- Temporal 的核心优势是 `wait_condition` —— 等用户回复时可以挂起不占资源，等一周都行，这是 LangGraph checkpointer 做不到的（需要手动 re-invoke）
- 当前不选的原因：部署太重（需要 Temporal Server + DB），LLM 流式输出需要额外 hack，单进程够用

**详细扩展**：
> 见 [top-知识扩展.md#temporal-分布式工作流引擎深入](top-知识扩展.md#temporal-分布式工作流引擎深入)

### Q4：ReAct 的最大问题是什么？为什么不适合费曼教学链路？

**来源阶段**：1.2

**用户回答**：
> 所有的状态都由 LLM 决定，可能产生动荡和流程漂移，而费曼的教学链路是一个相对固定流程链路，对一个主题要按照基本固定的流程教学和反馈。

**评价**：

| 论点 | 评价 | 说明 |
|---|---|---|
| LLM 自决策导致动荡和流程漂移 | ✅ 正确 | ReAct 核心问题——不可预测 |
| 费曼链路是固定流程，不适合 | ✅ 正确 | [graph_v2.py:104](../app/agent/graph_v2.py#L104) 的 `route_after_restate` 只有 2 个硬编码分支 |
| "相对固定" | ⚠️ 建议用"强约束" | 5 步顺序是硬编码的图拓扑，不是"相对"固定 |

**总分**：优秀（抓住核心矛盾：灵活性 vs 确定性）

**补充要点**：
- 用"强约束流程"代替"相对固定流程"——图拓扑是编译时确定的，不是运行时 LLM 选择的
- ReAct 适合的场景：开放式问答、研究助手；不适合的场景：有固定 SOP 的业务流程

**详细扩展**：
> 见 [top-知识扩展.md#react-vs-langgraph-流程可控性对比](top-知识扩展.md#react-vs-langgraph-流程可控性对比)
