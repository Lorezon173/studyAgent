# LearningAgent 学习路线 v2（费曼式 · 面试导向）

> 修订时间：2026-05-04
> 修订原因：用户明确学习目标 = 面试可讲清，需重点关注「设计选型 + 替代方案 + 适用场景」
> 学习方式：费曼式对话（讲解 → 启发 → 用户回答 → 评价纠偏 → 沉淀文档）

---

## 顶层学习目标（重要）

**最终能力检验：能在 1 小时面试中清晰讲完整个项目，并能回答以下三类问题。**

| 维度 | 目标 |
|---|---|
| **What** | 项目做什么、模块怎么分、调用链怎么走 |
| **How** | 关键技术（Agent 编排 / 并发 / RAG / 可观测）的具体实现细节 |
| **Why** | 为什么这么选型？还有哪些替代方案？各自适用场景？|

第三类（Why）是**关键**——光会描述代码不够，要能站到设计者位置反推决策链。

---

## 学习方式：费曼式对话

每讲遵循固定节奏：

```
1. 我用类比/直觉先讲一个小颗粒概念（不超过 1 个核心点）
2. 给 1-3 处代码证据锚点（不堆砌）
3. 抛出 2-3 个启发问题，要求你不查代码先答
4. 你回答后我做评价（指出对/错/错位/盲点）
5. 必要时补充正确版本，然后进入下一讲
6. 每个阶段结束，把"经过你纠偏后的理解"沉淀到 Learned/0X_*.md
```

**禁止行为**：直接堆文档、一次性塞太多代码、跳过"你回答"环节。

## 五阶段路线（按面试讲述顺序倒推）

### 阶段 1 · 项目主线总览（面试开场 5 分钟版）

**面试场景**：「介绍下你这个项目」

**学完后能回答**：
- 这是个什么项目？解决什么问题？
- 顶层数据流：用户输入 → 经过哪些关键模块 → 产生什么输出
- 5 个入口（FastAPI / CLI / Chainlit / Celery / SLO CLI）的关系
- 核心抽象：`LearningState` / 17 节点 / 双轨 RAG / 同异步双路径
- 关键 feature flag：`use_graph_v2` / `async_graph_enabled` / `langfuse_enabled` / `rag_enabled`

**讲法**：
- 1.1 费曼学习法 → 5 个 stage → 状态机直觉（已开始）
- 1.2 LearningState 是什么、为什么用 TypedDict 不用 Pydantic
- 1.3 17 节点的职责分组（教学 7 + QA 4 + 编排 6）
- 1.4 双 flag 切出来的 4 种运行模式
- 1.5 5 个入口最终都收口到哪里

**沉淀产出**：`Learned/02_项目主线总览.md`

---

### 阶段 2 · Agent 编排深挖（面试核心 15 分钟）

**面试场景**：「你的 Agent 是怎么编排的？为什么用 LangGraph？」

**学完后能回答**：
- LangGraph vs 手写状态机 vs LangChain Agent vs ReAct，**为什么选 LangGraph**
- 节点注册机制：`@node` 装饰器 + Registry 模式 vs 直接 `graph.add_node`，**为什么这么设计**
- 条件边 vs 固定边：在哪些场景必须用条件边
- Checkpoint 机制：SQLite checkpointer 怎么实现"断点续聊"
- Retry policy：节点级 vs 全局，AI 调用重试要避开什么坑（如幂等性）
- Replan 机制：什么时候触发？replan 节点为什么直接 → END？
- Recovery 节点：错误路由出口为什么共用一个 recovery？
- EvidenceGate + AnswerPolicy：为什么要拆成两个节点？

**讲法**（以"对比"为主）：
- 2.1 状态机演进：硬编码 if-else → V1 自实现 → V2 LangGraph
- 2.2 节点装饰器机制 + 替代方案对比
- 2.3 条件路由的 6 个 routers 各自的决策依据
- 2.4 Checkpoint 与会话恢复
- 2.5 重试与幂等性
- 2.6 错误恢复（Recovery）的"共用出口"模式

**沉淀产出**：`Learned/03_Agent编排.md`

**关键替代方案对比**（必须能答出）：

| 当前选型 | 替代方案 | 何时换更好 |
|---|---|---|
| LangGraph | 手写状态机 / LangChain Agent / AutoGen / CrewAI | 待沉淀 |
| TypedDict 状态 | Pydantic BaseModel / dataclass | 待沉淀 |
| 装饰器 + Registry | 直接 add_node / 配置文件驱动 | 待沉淀 |
| 节点级 retry | 全局 retry / try-except 内嵌 | 待沉淀 |
| SQLite checkpointer | Redis / Postgres / 内存 | 待沉淀 |

### 阶段 3 · RAG 实现深挖（面试核心 15 分钟）

**面试场景**：「你的 RAG 是怎么做的？为什么不用现成的？」

**学完后能回答**：
- BM25 + Dense + RRF + Rerank 四件套：每一件解决什么问题？少一件会怎样？
- RRF（Reciprocal Rank Fusion）的原理与公式（`1/(k+rank)`），vs 加权和、CombSUM
- Rerank：交叉编码 vs 双编码，**项目用的是哪种**
- 双轨存储（global + personal）：为什么不合并？user_id 隔离怎么做？
- Evidence Gate：覆盖度评估算法（关键词覆盖 vs 语义相似度）
- Answer Policy：什么时候出"边界声明"
- 图文统一检索：OCR 在 ingest 期还是 query 期？
- Tool Route：3 个工具技能（local_textbook / personal_memory / web_search）怎么决策路由？

**讲法**：
- 3.1 RAG 三层职责（rag_service / rag_coordinator / retriever）
- 3.2 召回阶段：BM25 + Dense 双路并行
- 3.3 融合阶段：RRF 公式 + 替代方案对比
- 3.4 Rerank：cross-encoder vs bi-encoder
- 3.5 Evidence Gate 与 Answer Policy 拆分
- 3.6 双轨存储与 user_id 隔离
- 3.7 工具路由决策

**沉淀产出**：`Learned/04_RAG实现.md`

**关键替代方案对比**：

| 当前选型 | 替代方案 | 何时换更好 |
|---|---|---|
| BM25+Dense+RRF | 纯向量 / 纯 BM25 / ColBERT | 待沉淀 |
| RRF 融合 | 加权和 / CombSUM / 学习排序 | 待沉淀 |
| simple rerank | cross-encoder / Cohere Rerank API | 待沉淀 |
| JSONL 文件存储 | 向量数据库（Milvus/Qdrant/PGVector） | 待沉淀 |
| 双轨（global+personal） | 单库 + 元数据过滤 | 待沉淀 |
| Evidence Gate 关键词覆盖 | 语义相似度阈值 / LLM Judge | 待沉淀 |

### 阶段 4 · 并发与异步架构（面试核心 10 分钟）

**面试场景**：「你的异步路径是怎么做的？为什么不直接用 async/await？」

**学完后能回答**：
- 同步 / 异步双路径设计的**真正动机**（不是性能！）
- Celery + Redis 的角色：Celery 跑 Graph，Redis 既做 broker 又做 pubsub
- 流式 token 怎么从 LLM 一路桥接到 SSE 客户端：4 步桥接链
- `progress_sink` 模式 vs Queue 模式 vs WebSocket
- 同步路径里的 `Queue + Thread` 为什么要这么写（FastAPI 同步 endpoint 内启动后台线程）
- Feature flag 切换的工程价值（灰度发布、回滚）
- 异步路径的失败模式（worker 超时 / Redis 断开 / 任务堆积）怎么处理

**讲法**：
- 4.1 同步路径：Queue + Thread + stream_to 上下文
- 4.2 异步路径：dispatch → Celery task → Redis pubsub → SSE
- 4.3 progress_sink 抽象的设计意图
- 4.4 ASYNC_GRAPH_ENABLED feature flag 的灰度价值
- 4.5 失败处理与超时

**沉淀产出**：`Learned/05_并发与异步.md`

**关键替代方案对比**：

| 当前选型 | 替代方案 | 何时换更好 |
|---|---|---|
| Celery + Redis | RQ / Dramatiq / Arq / 直接 asyncio | 待沉淀 |
| Redis pubsub | RabbitMQ / Kafka / NATS | 待沉淀 |
| SSE | WebSocket / gRPC streaming | 待沉淀 |
| 同步路径用 Queue+Thread | 全 async / 全 sync 阻塞 | 待沉淀 |

### 阶段 5 · 可观测与运维（面试加分 10 分钟）

**面试场景**：「你怎么保证这个 Agent 稳定？怎么发现问题？」

**学完后能回答**：
- Langfuse v4：trace / span / observation 三层概念
- 节点级 span 嵌套：session span → root span → 节点 span
- Payload 脱敏与截断（PII 治理）
- SLO 6 SLI 的具体口径（哪些是 latency 类、哪些是 quality 类）
- 12 题回归集的设计思路（为什么是 12 题而不是 100 题）
- Alert 三级（INFO/WARN/CRIT）的触发条件
- Runbook 的价值：值班场景化
- branch_trace 字段的设计（why 不直接读 Langfuse）

**讲法**：
- 5.1 三层监控：branch_trace（业务） + Langfuse（trace） + SLO（指标）
- 5.2 Langfuse v4 嵌套结构
- 5.3 SLO 6 SLI 口径
- 5.4 告警分级与 Runbook 联动
- 5.5 脱敏与截断

**沉淀产出**：`Learned/06_可观测与运维.md`

**关键替代方案对比**：

| 当前选型 | 替代方案 | 何时换更好 |
|---|---|---|
| Langfuse | Phoenix / LangSmith / 自建 OTel | 待沉淀 |
| 自建 SLO 门禁 | Grafana SLO / Sloth / Pyrra | 待沉淀 |
| branch_trace 字段 | 直接读 Langfuse / 结构化日志 | 待沉淀 |

## 互动协议（已升级）

每讲我会：
1. 用 1 个类比 + 1-3 处代码证据**讲清一个小颗粒**
2. 抛 2-3 个启发问题（**不允许查代码，凭直觉答**）
3. 你回答后，我做"对/错/错位/盲点"四象评价
4. 必要时补充正确版 + 替代方案对比
5. 进入下一讲

阶段结束时：
- 把"经过你纠偏后的理解"按你能讲清的程度沉淀到对应文档
- **文档不是讲义，而是你这一阶段学习的快照**

你随时可以用：
- **再举一个例子** / **我没懂**
- **跳过这个小点** / **回头到 X**
- **只讲为什么不讲怎么做** / **多对比替代方案**
- **测我一下**（我会出小测验）
