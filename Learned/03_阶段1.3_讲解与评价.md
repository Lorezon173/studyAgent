### 情景题 1：新来的实习生要加一个"实验题"节点

> 想在费曼教学流程里加一个"实验题"节点——讲完知识点后，先出一道编程题让用户做，做了再复述。放在哪个文件？在 explain 和 restate_check 之间插入，需要改哪几个地方？

**用户回答**：
> 放在 orchestration 文件里。不清楚你说的图是什么。

**评价**：

| 论点 | 评价 | 说明 |
|---|---|---|
| ❌ 放 orchestration | **概念错位** | orchestration 是"调度员"（路由、规划、错误恢复），不是"教师"。"实验题"属于教学环节，应该放在 teach.py |
| ⚠️ 不清楚图结构 | 合理反馈 | 已更新 ruler.md：后续提问涉及图/字段时必须先给出结构信息 |

**纠偏**：正确答案是 teach.py。"实验题"是教学过程的一个环节（类似 diagnose、explain），属于"教师"职责，不是"调度员"职责。

### 情景题 2：意图路由器的困境

> 用户输入："帮我复习一下上次学的哈希表，但是我还想问一下 Python 的 GIL 是什么。" 一句话同时包含"复习"和"直接问答"两种意图，怎么处理？

**用户回答**：
> 先进行检索，获取上次学习情况的总结，然后再进行到 qa，回答用户的问题。将两个需求独立看待，分别处理回答。

**评价**：

| 论点 | 评价 | 说明 |
|---|---|---|
| ✅ 拆成两个独立需求 | 方向正确 | 代码中 `intent_router_node` 确实区分不同 intent |
| ✅ 先复习再回答 | 合理的优先级判断 | `review` 和 `qa_direct` 是不同的路由分支 |
| ⚠️ 缺少对"只能选一个"的认知 | 需补强 | 当前 `intent_router_node` 只返回**一个**主 intent，不会并行处理 |

**核心认知**：当前设计的取舍是——一次只走一条路径。如果用户同时要两个，系统选主 intent，另一个就丢了。这是简化设计的代价。

---

### 情景题 3：错误恢复的共用出口

> 6 个条件路由都可能走到 `recovery_node`。为什么不直接在各节点里 try-except，而要设计一个共用 recovery 节点？

**用户回答**：
> 每个独立 try 之后，比较难追溯具体是哪个模块出了问题。统一 recovery 可以追溯整个执行链路。但统一 recovery 在总出错时，上下文需要独立清理和环境切换，否则可能互相干扰，环境频繁切换也会带来开销。

**评价**：

| 论点 | 评价 | 说明 |
|---|---|---|
| ✅ 统一出口利于追溯定位 | 正确 | 最核心的理由 |
| ✅ 独立 try-except 难追溯 | 正确 | 分散的错误处理让调试变成"找 needle in haystack" |
| ⚠️ 上下文互相干扰 / 环境切换开销 | **过度推论** | recovery_node 是无状态的，每次执行只依赖传入 state，不存在"上下文残留" |

**亮点**：抓住了解耦和可追溯性的核心。"上下文干扰"的担心虽不成立（recovery_node 无共享可变状态），但这种"预判副作用"的思考习惯很好——在多线程/分布式场景下确实会成立。

## 二、正式讲解：17 节点职责分组与编排骨架

### 核心概念

**一句话**：17 个节点分三组——教学组（teach，7 个）负责费曼 5 步，问答组（qa，4 个）负责知识检索和回答，编排组（orchestration，6 个）负责"往哪走"。

**类比**：

```
一所学校
├ 教师（teach）      → 负责"教什么、怎么教"
├ 图书馆员（qa）     → 负责"找资料、回答问题"
└ 教务处（orchestration）→ 负责"排课表、处理异常、决定下一步"
```

---

### 三组节点完整清单

**教学组** — [app/agent/nodes/teach.py](../app/agent/nodes/teach.py)（7 个）

```
├ history_check_node           检查学生有没有学过这个主题
├ ask_review_or_continue_node  学过？问学生要复习还是继续
├ diagnose_node                第1步：诊断学生已有认知
├ explain_node                 第2步：讲解
├ restate_check_node           第3步：让学生复述，评估掌握度
├ followup_node                第4步：追问补救
└ summarize_node               第5步：总结 + 复习建议
```

**问答组** — [app/agent/nodes/qa.py](../app/agent/nodes/qa.py)（4 个）

```
├ rag_first_node           先去知识库搜

├ llm_answer_node          没搜到 / 不需要搜，用 LLM 直接答

```

**编排组** — [app/agent/nodes/orchestration.py](../app/agent/nodes/orchestration.py)（6 个）

```
├ intent_router_node       判断用户想干嘛（学 / 问 / 复习 / 换主题）
├ replan_node              用户换主题 / 纠偏时重新规划
├ retrieval_planner_node   制定检索策略（用什么方法搜）
├ evidence_gate_node       检索到的证据够不够？够 → 回答，不够 → recovery
├ answer_policy_node       证据够了，选哪种回答模板
└ recovery_node            统一错误恢复出口
```

### 每个节点的"读什么 / 写什么"

**教学节点**：

| 节点 | 读取 | 写入 | stage 变化 |
|---|---|---|---|
| `history_check_node` | `topic` | `has_history`, `history_summary` | — |
| `diagnose_node` | `user_input` | `diagnosis` | → `"diagnosed"` |
| `explain_node` | `diagnosis`, `topic` | `explanation` | → `"explained"` |
| `restate_check_node` | `explanation`, `user_input` | `restatement_eval`, `mastery_*` | → `"restatement_checked"` |
| `followup_node` | `mastery_*`, `error_labels` | `followup_question` | → `"followup_generated"` |
| `summarize_node` | 全部教学字段 | `summary` | → `"summarized"` |

**编排节点**：

| 节点 | 读取 | 写入 |
|---|---|---|
| `intent_router_node` | `user_input` | `intent`, `intent_confidence` |
| `replan_node` | `user_input`, `topic` | `current_plan`, `need_replan` |
| `evidence_gate_node` | `rag_context` | `gate_status`, `gate_coverage_score` |
| `recovery_node` | `node_error`, `error_code` | `recovery_action`, `fallback_triggered`, `reply` |

**关键规则**：每个节点只写自己那组字段，不跨组写入。物理分组 → 逻辑隔离。
