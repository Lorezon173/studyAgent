# P0 稳定性收口计划（当前阶段）

## 目标

在已完成 Agent 解耦、RAG 双轨、检索升级与 Tool 路由的基础上，进行一轮“稳定性收口”，确保后续 P1/P2 改造建立在可回归、可观测、可审计的基线上。

## 范围

- 质量回归：核心链路（chat / knowledge / profile / auth / skills）
- 多用户隔离：`user_id` 与 personal scope 的越权与串号防护
- 工具链路稳定性：tool_route 与 tool_executor 执行一致性
- 观测留痕：关键分支日志字段完整性（intent/tool_route/citations）

## 执行计划

### 1. 回归矩阵与基线锁定

- [ ] 建立回归清单：按模块列出关键测试文件与通过标准
- [ ] 固化本地回归命令（快速集 + 全量集）
- [ ] 对当前分支做基线快照（测试结果与关键行为截图/样例）

### 2. 多租户与权限边界

- [ ] 补充 personal 检索越权测试（A 用户不可读 B 用户）
- [ ] 补充会话绑定测试（会话与 user_id 绑定后不可串号）
- [ ] 补充 profile 读路径的 user_id 隔离断言

### 3. 工具执行链路稳定性

- [ ] 校验 tool_route 选择与 tool_executor 实际执行一致
- [ ] 补充“工具无结果 -> 兼容回退”测试
- [ ] 校验 citations 中 `scope/user_id/tool` 元数据完整

### 4. 错误处理与降级策略

- [ ] 明确并测试 LLM/检索异常时的降级路径
- [ ] 统一错误返回语义（400/500 边界）
- [ ] 避免 silent failure，关键失败点写入 trace

### 5. 文档与发布门禁

- [ ] 更新 README 与 plan/README 的当前状态描述
- [ ] 对齐 worklog 与测试导航索引
- [ ] 形成 P0 完成判定清单（进入 P1 的准入条件）

## 验收标准

- 关键回归测试通过（核心 API、RAG、Agent、Skills）
- 多用户隔离与会话绑定用例通过
- 工具路由与执行链路可追踪（state + branch_trace）
- 文档状态与实际代码一致

## 风险与缓解

- 风险：工具链路改造导致旧测试失效  
  缓解：保留兼容回退路径，分批替换测试桩

- 风险：跨模块改动引入隐性回归  
  缓解：以“最小改动+分层测试”推进，每步都可回滚

- 风险：环境差异导致 CI 与本地不一致  
  缓解：固定回归命令与依赖版本，记录失败样例
