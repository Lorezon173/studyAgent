本文基于以下两部分对照得出：

- 参考文档：`plan/架构演进.md`
- 实际代码：`app/services/agent_service.py`、`app/services/agent_runtime.py`、`app/services/learning_analysis.py`、`app/services/rag_store.py`、`app/api/profile.py` 等

---

## 1. 结论摘要

`架构演进.md` 的方向 **整体值得参考**，尤其是：

- 解耦 `agent_service` 巨石编排
- LLM-as-a-Judge 结构化评估
- RAG 纯检索定位
- MCP Tool-Calling 演进路径

但该文档是“目标架构蓝图”，直接照搬会与当前实现产生断层。建议采用“**低风险渐进改造**”：先保行为、再拆职责、最后上 ToolNode/MCP。

## 2. 当前项目实际状态（代码事实）

### 2.1 已具备基础

- 三阶段子图已稳定：`initial_graph / restate_graph / summary_graph`
- 意图路由已有：`teach_loop / qa_direct / review / replan`

### 2.2 当前主要瓶颈

- `AgentService.run` 仍承担过多职责（上下文构建 + 路由 + 图调度 + 落盘）

- `qa_direct` 目前是分支逻辑，不是 LangGraph 子图（恢复流程依赖手写代码）

## 3. 对 `架构演进.md` 各项建议的适配分析

## 3.1 1.1 巨石服务拆解 —— **高价值，立即执行**

原建议合理。当前可落地方式：

1. 新建 `app/services/orchestration/` 子模块：
   - `context_builder.py`
   - `stage_orchestrator.py`
   - `persistence_coordinator.py`
2. 首轮仅“搬代码不改行为”，保持 `AgentService.run` 的外部接口不变。
3. 第二轮再将 `run` 收敛为 orchestrator 门面。

风险控制：每拆一块就跑现有 `chat/profile/session` 测试，避免一次性大重构。

## 3.2 1.2 LLM-as-a-Judge —— **值得做，但需双轨评估**

原建议正确，但不建议一步替换规则评估。  
建议改为：

- 新增 `evaluator_node`（或 `evaluation_service`）
- LLM 输出严格 schema：`mastery_score_1to5`、`error_labels`、`confidence`、`rationale`
- 规则评估暂时保留为 fallback（当 LLM 异常/JSON 解析失败时）

验收标准：

- 结构化输出成功率 > 95%
- 与人工标注样本一致性显著高于当前规则法

## 3.3 1.3 柔性状态流转（ToolNode/Sub-graph）—— **推荐分阶段做**

当前实现已支持 `qa_direct` 分支，但仍在 `AgentService` 里手写控制。  
建议演进：

1. 先引入 `qa_subgraph`（不一定马上上 ToolNode）
2. 将“答疑后恢复原 stage”状态恢复逻辑移入图节点
3. 再升级到 LangGraph ToolNode 统一调度

这样能减少“中途答疑打断教学链路”的维护复杂度。

## 3.4 2.1 RAG 纯检索定位 —— **已基本符合，应继续强化**

你当前实现已经是“检索返回证据 + 注入 prompt”的模式，没有把 RAG 框架当编排器使用，这点与文档一致。  
建议继续：

- 将检索接口统一抽象为 `Retriever` 协议
- 节点只消费 `Document[]`，不关心底层存储和模型

## 3.5 2.2 双轨记忆（Global / Personal）—— **当前未完成，优先级高**

当前问题：主要按 `topic` 聚合，没有 `user_id` 强隔离。  
建议：

- 增加 `user_id` 到会话、个人记忆、personal RAG 写入与检索接口
- personal 检索必须带 metadata filter（硬约束）
- global 知识库保持只读，不允许用户写入

这是多用户场景下必须先补的安全边界。

## 3.6 3. 向量技术选型 —— **建议“接口先行，底座后换”**

文档给出的“原生向量 SDK 优先”是可行的。  
结合你当前代码，推荐先做：

- 定义 `VectorStoreAdapter` 接口
- 现有 JSONL/simple 实现作为 `LocalAdapter`
- 新增 `QdrantAdapter` 或 `ChromaAdapter`（二选一）

不要直接把业务代码绑死到某个 SDK。

## 3.7 4. MCP 演进 —— **方向正确，但当前不宜立刻微服务化**

现阶段先做 Tool-Calling 抽象，暂不急于“独立 MCP Server 全拆”。  
建议顺序：

1. 先在单体内把检索入口标准化为 tool（`search_local_textbook`/`search_personal_memory`）
2. 再接入外部 web search tool
3. 最后将工具逐步外移为 MCP 服务

否则会在稳定性与调试链路上付出过高成本。

## 4. 针对当前项目的分阶段落地计划（建议版）

## Phase A（低风险重构，1个迭代）

- 拆 `AgentService` 为三协调器（保持行为不变）
- 提取 `Retriever`、`Evaluator` 接口（先不替换实现）
- 增加 `user_id` 字段贯穿请求/会话（先存储，不启用强校验）

交付物：

- 新目录结构与模块边界
- 回归测试全绿

当前进展（已开始）：

- 已落地 `ContextBuilder / StageOrchestrator / PersistenceCoordinator` 代码骨架，并接入 `agent_service` 主流程。
- 当前目标仍是“行为兼容优先”，后续继续推进 `Evaluator` 与 `ToolNode`。
- 已接入 LLM-as-a-Judge 的结构化评估链路（含规则 fallback），进入可灰度验证阶段。

## Phase B（质量提升，1个迭代）

- 新增 `evaluator_node`，启用 LLM 结构化评估
- 保留规则 fallback，增加评估对比日志
- `qa_subgraph` 落地，保证答疑后恢复阶段

交付物：

- 评估结构化结果入库
- 中途答疑连续性测试通过

## Phase C（RAG治理，1个迭代）

- 上线双轨记忆：global/personal 物理或逻辑分区
- personal 强制 `user_id` 过滤
- 增加检索审计字段（source_type, user_scope, filter_applied）

交付物：

- 多租户隔离测试通过
- 检索路径可审计

当前进展（更新）：

- 已在 `knowledge ingest/retrieve` 接口增加 `scope/user_id` 契约。
- 已在 `rag_store` 落地强隔离：
  - `global` 仅检索全局轨道；
  - `personal` 强制 `scope=personal AND user_id=当前用户`。
- 已在聊天链路接入 `user_id` 并融合双轨证据（global + personal），引用中返回 `scope/user_id` 元数据。

## Phase D（MCP 演进，后续）

- ToolNode 接入统一检索工具
- 外部 Web Search 工具接入
- 再评估拆分独立 MCP Server

---

## 5. 本项目当前不建议立即做的事

- 直接大规模改成“全 MCP 微服务”
- 一次性替换全部规则评估为 LLM（无 fallback）
- 先换数据库/向量底座再做接口抽象

这些都会显著增加回归风险。

---

## 6. 建议优先级（执行顺序）

1. **P0**：`agent_service` 解耦 + `user_id` 贯穿字段  
2. **P1**：`evaluator_node`（LLM结构化 + fallback）  
3. **P1**：`qa_subgraph`（柔性流转）  
4. **P2**：双轨记忆强隔离（metadata filter）  
5. **P3**：ToolNode 与 MCP 外移
