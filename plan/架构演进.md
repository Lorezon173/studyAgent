Learning Agent 架构演进与重构指南 (v0.2 - MCP 阶段)
本指南汇总了针对基于“费曼学习法”的 LangGraph Agent 项目的架构审查与重构建议。文档结构经过优化，旨在为编程 Agent 或开发者提供无歧义、强逻辑、可执行的迭代蓝图。

1. Agent 核心编排机制优化
针对当前 FastAPI + LangGraph 架构中存在的职责耦合与规则脆弱性，建议进行以下重构：

1.1 拆解“巨石”服务 (agent_service.py)
现状缺陷：agent_service.run 承担了意图路由、上下文构建、图调度和持久化，违反单一职责原则。

执行建议：采用中间件/责任链模式重构。拆分为独立的类或模块：

ContextBuilder：专职构建 {topic_context} 和 {long_term_context}。

StageOrchestrator：专职管理 LangGraph 的阶段调用（initial_graph / restate_graph / summary_graph）。

PersistenceCoordinator：专职负责 SQLite 和 RAG 的数据落盘操作。

1.2 升级评估机制 (LLM-as-a-Judge)
现状缺陷：依赖启发式规则和关键词匹配提取掌握度分数与错因，易受用户文本风格干扰。

执行建议：在 summarize_node 之后引入 evaluator_node。利用大模型原生结构化输出（如 with_structured_output）直接提取用户的学习状态。

输出定义：强制 LLM 输出严格的 JSON Schema（包含 mastery_score 1-5分，以及 error_labels 数组）。

1.3 引入柔性状态流转 (Tool Node / Sub-graph)
现状缺陷：用户在“复述”或“追问”阶段提问会导致强制 Fallback 到 initial_graph，中断教学连续性。

执行建议：利用 LangGraph 的 Tool Node 或 Sub-graph 机制。当检测到“答疑意图”时，切入 qa_direct 子图进行解答，完成后恢复原阶段状态，而非重置整个教学流。

2. RAG 机制与双轨制记忆架构
为防止大模型幻觉与多用户记忆污染，RAG 必须被降级为纯粹的“检索器（Retriever）”，并实现物理与逻辑的数据隔离。

2.1 RAG 定位：纯检索组件
架构规范：不要使用带有编排能力的 RAG 框架（如 LlamaIndex 的 QueryEngine），避免与 LangGraph 争夺控制权。

数据流向：调用检索 API -> 获取 [Document, Document...] -> 拼接为字符串 -> 注入 Prompt 插槽 {retrieved_knowledge} -> 由 LangGraph 节点统一调用 LLM 生成。

2.2 双轨制记忆设计 (Dual-Track Memory)
轨道一：全局客观知识库 (Global Knowledge RAG)

存储内容：书籍、教材、官方文档（静态、客观）。

访问权限：全局共享，只读不写。

检索策略：基于 Topic 或 Query 的纯语义相似度检索。

轨道二：用户私域记忆库 (User Personal Memory RAG)

存储内容：用户历史对话的非结构化原话片段（动态、主观）。

访问机制：强制多租户隔离。在写入和检索向量数据库时，必须使用元数据过滤（Metadata Filtering），例如 filters=[ExactMatchFilter(key="user_id", value="user_001")]。

3. RAG 框架技术选型
基于纯检索需求与 Python 技术栈，框架选型优先级如下：

原生向量数据库 SDK (最优选)：直接使用 ChromaDB、Qdrant 或 Milvus 的 Python Client 配合 sentence-transformers。极致轻量，代码完全可控，无黑盒流转。

LangChain Core/Community (次优选)：利用 VectorStoreRetriever 标准接口包装底层数据库。与当前项目生态契合度极高，便于快速切换不同向量数据库。

避坑指南：强烈建议弃用 Embedchain（黑盒封装过重）以及纯粹为了检索而引入 Haystack（Pipeline 编排概念与 LangGraph 重叠）。

4. MCP (Model Context Protocol) 演进蓝图
为未来接入外部搜索（联网、内部文档等），架构需从“静态拼装”向“统一工具调用（Tool-Calling）”演进。

4.1 核心理念转换：Tool-Calling 范式
机制描述：废弃在 agent_service 中强制查询所有上下文并塞入 Prompt 的做法。将所有信息获取渠道封装为标准的 Tool（如 search_web_mcp, search_local_textbook, search_personal_memory）。

调度方式：在 LangGraph 中增加 ToolNode，使 Agent 根据用户当前输入，自主决策调用何种检索工具。

4.2 独立 MCP Server 部署
架构解耦：将本地书籍 RAG 和个人记忆 RAG 从主控 FastAPI 服务中剥离，封装为独立的微服务，并对外暴露标准的 MCP 协议接口。

收益：主控代码仅保留状态机与决策逻辑，底层知识检索逻辑实现跨语言、跨框架的完全隔离。

4.3 知识路由与冲突解决 (Knowledge Router)
冲突解决规则：在 EXPLAIN_PROMPT 中硬编码知识采信优先级（例如：本地教材权威性 > 外部网络搜索 > 用户个人记忆）。

路由机制：在执行核心检索前，增加轻量级意图分类器，将不同性质的 Query 路由至指定的 MCP 工具（例如，前沿资讯路由至 Web Search，基础概念路由至 Local Textbook）。

4.4 迭代演进路线 (Roadmap)
v0.1 (当前基线)：固定 Prompt 模板 + Memory/SQLite 状态流转。

v0.2 (RAG 引入)：接入轻量级原生向量检索函数，将结果作为字符串拼接至 Prompt 模板；落实双轨制记忆隔离。

v0.3 (MCP 改造)：外部搜索与本地 RAG 统一抽象为 Tools，引入 LangGraph ToolNode，实现模型自主的数据源检索与知识整合。
