## 1. 项目概述

### 1.1 项目定位

LearningAgent 是一个基于 **LangGraph 状态图编排** 的智能学习代理系统，创新性地将 **费曼学习法** 转化为可执行的多轮对话教学闭环。系统支持：

- 🎯 **个性化学习路径规划** - 基于诊断的自适应教学内容生成
- 📚 **RAG 知识检索增强** - BM25 + 向量 + Rerank 混合检索
- 📊 **学习进度追踪** - 掌握度评估与复习建议
- 🔍 **全链路可观测性** - Langfuse 追踪 LLM/RAG/Tool 执行
- ⚡ **异步任务队列** - Celery + Redis 支持 10-50 并发用户

### 1.2 核心价值

| 维度 | 实现方式 | 业务价值 |
|------|----------|----------|
| 教学质量 | 费曼学习法闭环 (诊断→讲解→复述→追问→总结) | 科学验证的学习方法 |
| 知识准确性 | RAG 混合检索 + 引用溯源 | 减少 LLM 幻觉，提供可靠知识 |
| 用户体验 | SSE 流式响应 + 会话恢复 | 实时反馈，无缝续学 |
| 系统可靠性 | 重试策略 + 检查点持久化 | 容错能力强，状态不丢失 |
| 高并发支持 | Celery 任务队列 + Redis Pub/Sub | 解耦 HTTP 与 Agent 执行 |

## 3. 核心技术亮点

### 3.1 Celery + Redis 任务队列 (⭐ 新增)

**问题：** HTTP 请求同步等待 LangGraph 执行（10s+），导致连接池枯竭，多用户访问时请求排队。

**解决方案：** Celery 任务队列 + Redis Pub/Sub 实现异步执行与实时进度推送

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ FastAPI     │────▶│   Redis     │────▶│   Celery    │
│ /chat/async │     │ Task Queue  │     │   Worker    │
└─────────────┘     └─────────────┘     └──────┬──────┘
       │                                       │
       │         ┌─────────────┐               │
       └────────▶│ Redis PubSub│◀──────────────┘
                 │  Progress   │
                 └─────────────┘
                       │
                       ▼
                 ┌─────────────┐
                 │ SSE Stream  │
                 │ /chat/stream│
                 └─────────────┘
```

**核心实现：**

```python
# 任务定义 (app/worker/tasks.py)
@celery_app.task(bind=True, max_retries=3, retry_backoff=2)
def run_agent_task(self, session_id: str, topic: str | None, 
                   user_input: str, user_id: int | None = None):
    """异步执行 Agent 任务，推送进度到 Redis"""
    reporter = ProgressReporter(task_id=self.request.id)
    
    result = agent_service.run(
        session_id=session_id,
        topic=topic,
        user_input=user_input,
        user_id=user_id,
        on_progress=lambda event, payload: reporter.publish(event, payload),
    )
    return result

# 进度报告 (app/worker/progress.py)
class ProgressReporter:
    def publish(self, event: str, payload: dict) -> None:
        """发布进度到 Redis Pub/Sub"""
        message = {"event": event, "payload": payload}
        self._client.publish(f"task:{self.task_id}", json.dumps(message))
```

**技术成果：**
- ✅ 解耦 HTTP 请求与 Agent 执行
- ✅ 支持 10-50 并发用户
- ✅ SSE 流式响应实时推送进度
- ✅ 优雅降级：Redis 不可用时回退同步模式
- ✅ 自动重试：可配置重试次数和指数退避

**新增 API 端点：**

| 端点 | 方法 | 说明 |
|------|------|------|
| `/chat` | POST | 根据 `ASYNC_CHAT_ENABLED` 自动选择同步/异步 |
| `/chat/sync` | POST | 强制同步执行 |
| `/chat/status/{task_id}` | GET | 查询任务状态 |
| `/chat/stream/{task_id}` | GET | SSE 流式响应 |

### 3.2 LangGraph V2 状态图编排 (⭐ 核心创新)

**问题：** 传统多轮对话难以管理复杂状态流转，代码耦合度高。

**解决方案：** 基于 LangGraph 构建状态图，实现：

```python
# 6 个条件边路由函数，动态决策分支
graph.add_conditional_edges(
    "intent_router",
    route_by_intent,  # 根据意图路由
    {
        "history_check": "history_check",  # 教学模式
        "rag_first": "rag_first",          # 问答模式
        "replan": "replan",                # 重规划
        "summary": "summary",              # 复习模式
    }
)
```

**技术成果：**
- ✅ 13 个业务节点，6 个路由函数
- ✅ SQLite 检查点持久化，会话状态恢复
- ✅ 节点级重试策略 (LLM_RETRY, RAG_RETRY, DB_RETRY)
- ✅ 状态字段 30+ 个，支持历史检查/RAG 上下文/降级标记

### 3.3 RAG 混合检索引擎

**问题：** 单一检索方式召回率低，无法兼顾精确匹配与语义理解。

**解决方案：** 三路召回 + RRF 融合 + Rerank 重排

```
Query ──┬──▶ BM25 检索 (精确匹配) ──┐
        ├──▶ 向量检索 (语义理解) ──┼──▶ RRF 融合 ──▶ Rerank ──▶ Top-K
        └──▶ (可选) Web 搜索    ──┘
```

**技术成果：**
- ✅ 中文 2-gram/3-gram 分词优化
- ✅ 句子边界感知的文本分块
- ✅ RRF (Reciprocal Rank Fusion) 算法实现
- ✅ 支持全局/个人知识库隔离

### 3.4 全链路可观测性 (Langfuse 集成)

**问题：** LLM 应用黑盒运行，难以追踪调用链、排查问题。

**解决方案：** 装饰器模式 + Trace/Span 层级追踪

```python
# 装饰器模式，零侵入集成
@trace_llm("invoke")
def invoke(self, system_prompt: str, user_prompt: str) -> str:
    return self._get_llm().invoke(messages)

@trace_rag("retrieve")
def retrieve_knowledge_by_scope(query: str) -> list[dict]:
    return _hybrid_search(query)
```

**追踪数据：**
- 📊 LLM 调用延迟、Token 消耗
- 📊 RAG 检索命中率、Top-K 分数
- 📊 Tool 执行状态、错误日志
- 🔒 用户 ID 哈希脱敏 (SHA256)

### 3.5 容错与重试机制

**问题：** LLM API 不稳定，网络超时导致服务中断。

**解决方案：** 三层重试策略 + 指数退避

```python
# 节点级重试策略
LLM_RETRY = RetryPolicy(max_attempts=3, initial_interval=0.5, backoff_factor=2)
RAG_RETRY = RetryPolicy(max_attempts=2, initial_interval=0.2)
DB_RETRY   = RetryPolicy(max_attempts=2, initial_interval=0.1)

# Celery 任务重试
@celery_app.task(
    bind=True, 
    max_retries=3, 
    retry_backoff=2,
    autoretry_for=(ConnectionError, TimeoutError),
)
def run_agent_task(self, ...):
    ...
```
